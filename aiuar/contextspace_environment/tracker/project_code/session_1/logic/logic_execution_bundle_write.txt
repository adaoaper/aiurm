L1. Execution bundle write policy
- Receive *data_execution_bundle from the calling business project.
- *data_execution_bundle must contain:
    origin_contextspace   : contextspace of the calling project
    origin_entity         : entity of the calling project
    origin_project        : project name of the calling project
    origin_session        : session of the calling project
    execution_id          : execution identifier of the calling project
    timestamp             : YYYYMMDD_HHMMSS
    bundle_name           : descriptive name of the execution bundle
    bundle_root_path      : root folder path of the bundle
    bundle_manifest       : manifest or summary of included artifacts
    content               : optional bundle-level description or serialized manifest content
- The primary objective of the AI remains resolution of the calling governance Result steps and Apply contracts.
- Any script, helper, test, temporary file, note, manifest, or auxiliary artifact is instrumental only.
- The AI may organize the internal structure of the execution bundle freely.
- Internal file names, subfolders, scripts, tests, temporary files, and helper artifacts are at the AI's discretion.
- This freedom applies only inside the execution bundle.
- The AI MUST NOT create execution artifacts at aiuar_root or outside the resolved execution bundle.
- The execution bundle stores the relevant operational artifacts used or generated by the AI to resolve the governance.
- The bundle does not redefine governance execution as software production.
- Write one result artifact per execution bundle.
- Never overwrite an existing bundle result. Use execution identity and timestamp to preserve history.
- This write obligation remains in force even if the calling pipeline later aborts, fails validation, or emits no business Result artifact.
- Emit the result artifact with full bundle metadata.
